Section 01

Data Models

KEY TERM

Data Model

A set of concepts describing the structure of a database and the constraints it must obey. Think of it as the blueprint language for a database.

KEY TERM

Data Model Operation

Operations for specifying database retrievals and updates. Can be basic (built-in) or user-defined.

Categories of Data Models

High-Level

Conceptual / Semantic

Close to how humans perceive data. Uses entities, attributes, and relationships.

Example: Entity-Relationship (ER) Model

Mid-Level

Implementation / Representational

Balances user views with computer storage details.

Examples: Relational, Network, Hierarchical

Low-Level

Physical / Internal

Describes how data is physically stored in the computer.

Examples: Tree structures, Graphs, B+-trees

⚠ COMMON PITFALL

Students often confuse the ER Model (conceptual — for design) with the Relational Model (implementation — for actual database). ER diagrams are the design tool; relational tables are the implementation.

History of Data Models

1964–65 — Network Model Graphs

Implemented by Honeywell (IDS System). Adopted by CODASYL 1971. Data stored as interconnected graphs. Used in IDMS, DMS 1100, IMAGE.

~1965 — Hierarchical Model Trees

IBM + North American Rockwell. Led to IMS family. Data structured as hierarchical trees — parent/child relationships only.

1970 — Relational Model Tables

Proposed by E.F. Codd (IBM). First commercial system 1981-82. Now dominant — Oracle, DB2, SQL Server, PostgreSQL.

1980s–90s — Object-Oriented Model Objects

Persistent O-O languages: C++ (OBJECTSTORE), Smalltalk (GEMSTONE). Also O2, ORION (ITASCA), IRIS.

Recent — Object-Relational Model Hybrid

Started with Informix Universal Server. Exemplified in Oracle 10g, DB2, SQL Server. Combines OOP concepts with relational tables.

Relational Model — Key Concepts

Relation = Table of Values

A Relation is a mathematical concept based on sets
A relation can be thought of as a set of rows (tuples)
Each column is called an Attribute
Each row can be given a unique identifier (key)

Example: STUDENT Table

Name StudentNumber Class Major
Smith171CS
Brown82CS

← Each row is a tuple; each column is an attribute.

💡 ENTITY-RELATIONSHIP (ER) MODEL

Proposed by P. Chen (1976). A conceptual modeling tool for database requirements. Basic building blocks: Entities (real-world objects) and Relationships (associations between entities). Well-suited for relational database development.

Note: ER model did NOT originally include inheritance.

Section 02

Schema

KEY TERM

Database Schema

The description of a database — its structure and constraints. Also called the database intension. Changes infrequently.

KEY TERM

Database State / Snapshot

The actual data stored at a particular moment. Also called extension. Changes every time data is updated.

KEY TERM

Schema Diagram

A diagrammatic display of some aspects of a database schema — shows table names and attributes without actual data.

⚠ PITFALL

Schema vs State Confusion

Schema = blueprint (stable, rarely changes). State = actual data (changes constantly). Don't confuse structure with content!

Three-Schema Architecture

Proposed to support program-data independence and multiple views of data. Defines three distinct levels of abstraction:

EXTERNAL LEVEL — User Views

External Schema 1

User view A

External Schema 2

User view B

↕ external/conceptual mapping

CONCEPTUAL SCHEMA

Logical structure & constraints

↕ conceptual/internal mapping

INTERNAL SCHEMA

Physical storage structures

↓

STORED DATABASE

1

Internal Schema (Physical Level)

Describes how data is physically stored — file structures, indexes, access paths. Uses a physical data model (B+-trees, hash indexes, block sizes).

-- Example Internal Schema elements:
Cluster Index on SSN;
B+-tree on Age;
Block size = 100;
2

Conceptual Schema (Logical Level)

Describes the entire database structure and constraints for all users. Hides physical storage details. Focuses on what data means.

-- Example Conceptual Schema:
EMPLOYEE(SSN, EN, DNO, SALARY, EXP_YEAR, BDATE)
CUSTOMER(Name, Addr, Sex, Age)
3

External Schema (View Level)

Describes user-specific views derived from the conceptual schema. Different user groups see different subsets.

-- Derive a user view from conceptual:
CREATE MALE_CUSTOMER(Name, Addr)
  AS VIEW OF CUSTOMER(Name, Addr, Sex, Age)
  WHERE Sex = 'M';

Query: "List employees with more than 5 years experience"

-- External Schema (what users query):
SELECT e.ENAME, e.DEPT, e.EXP
FROM   EMP e
WHERE  e.EXP > 5;

-- This view is derived from the Conceptual Schema:
EMPLOYEE(SSN, EN, DNO, SALARY, EXP_YEAR, BDATE)

-- Which sits on the Internal Schema:
Cluster Index on SSN;
B+-tree indexes on DNO, EXP_YEAR, STARTDATE;

Data Independence

Logical Data Independence

Ability to change the conceptual schema without affecting external schemas or application programs.

External schemas unchanged
← — — — LOGICAL boundary — — —
Conceptual schema can change

Physical Data Independence

Ability to change the internal schema without affecting the conceptual schema.

Conceptual schema unchanged
← — — PHYSICAL boundary — — —
Internal schema can change

💡 MENTAL MODEL

Think of it like a building: the External schema = what tenants see (rooms/offices), Conceptual schema = the architectural floor plan, Internal schema = the electrical wiring and plumbing. Changing the plumbing doesn't change the floor plan or what tenants see.

Section 03

DBMS Components

DBMS Languages

DDL Data Definition Language

Used to define the schema — structure and constraints. Used by DBA and database designers.

CREATE ALTER DROP

Also: SDL (Storage Def Language) for internal schema; VDL (View Def Language) for external schemas.

DML Data Manipulation Language

Used to retrieve and update data. Can be embedded in host languages (COBOL, C) or used standalone.

SELECT INSERT UPDATE DELETE

HIGH-LEVEL DML (Non-procedural)

Specifies what data to retrieve, not how. Set-oriented. Also called declarative language. Example: SQL.

LOW-LEVEL DML (Procedural)

Record-at-a-time. Specifies how to retrieve data. Must be embedded in a programming language.

Main DBMS Modules

DDL Compiler

Processes DDL statements; updates the system catalog.

DML Compiler

Compiles DML statements embedded in host programs.

Query Compiler

Handles ad-hoc (interactive) queries from casual users.

Run-time Database Processor

Executes privileged commands and compiled transactions.

Stored Data Manager

Controls access to DBMS information on disk.

Concurrency/Backup/Recovery

Ensures ACID properties and fault tolerance.

Algorithm: How a Query is Processed

Step-by-step walkthrough of query processing in a DBMS:

Step 1 / 5

STEP 1 — User Submits Query

A casual user types an interactive query, OR an application program contains embedded DML commands.

-- User types this interactive query:
SELECT e.ENAME, e.EXP
FROM   EMP e
WHERE  e.EXP > 5;

STEP 2 — Query / DML Compiler

The Query Compiler parses the SQL, checks against the System Catalog/Data Dictionary to verify table/column existence and user permissions.

-- Compiler checks System Catalog:
-- Does table "EMP" exist? ✓
-- Does column "EXP" exist? ✓
-- Does user have SELECT permission? ✓
-- Generates query execution plan...

STEP 3 — Run-time Database Processor

The compiled query is passed to the Runtime DB Processor for execution. It coordinates with the Stored Data Manager.

-- Runtime processor receives optimized plan:
-- Plan: Use B+-tree index on EXP column
-- Scan index for EXP > 5
-- Retrieve matching tuples...

STEP 4 — Stored Data Manager

The Stored Data Manager accesses the physical database via OS I/O operations, using the Internal Schema's index structures.

-- Stored Data Manager executes disk I/O:
-- Read index blocks from disk
-- Fetch data pages for matching records
-- Buffer management (cache hot pages)

STEP 5 — Results Returned

Retrieved tuples are formatted according to the External Schema (user's view) and returned as the query result.

-- Result returned to user:
ENAME   | EXP
--------|-----
Alice   | 7
Bob     | 10
Charlie | 6

Section 04

DBMS Architecture

Centralized DBMS

Everything in a single system — DBMS software, hardware, applications, and user interfaces all combined.

[PC / Mainframe]
└── DBMS
└── App Programs
└── UI Software

2-Tier Client-Server

Client runs UI and application. Server runs the DBMS. Connected via ODBC (Open DB Connectivity) API.

[Client]
└── UI + App logic
── ODBC/Network ──
[DBMS Server]

3-Tier Client-Server

Most common for web applications. Intermediate Application Server stores business logic between client and DB.

[Client] GUI/Web
── HTTP ──
[App Server] Logic
── SQL ──
[DBMS Server]

💡 THREE-TIER SECURITY FEATURE

The three-tier architecture adds security by encrypting data at the server before transmission and decrypting at the client. The app server acts as a firewall between users and the raw database.

Q1

Which category of data model is closest to how users perceive data and uses concepts like entities and relationships?

Select the best answer.

Correct! Conceptual models like ER are designed to be human-friendly — close to how people think about real-world data.
Not quite. Physical models deal with storage; Implementation models bridge the two. The answer is B — Conceptual.

How to solve this:

The slide explicitly defines three categories: Conceptual (high-level, semantic), Physical (low-level, internal), and Implementation (representational).

The key phrase in the question is "closest to how users perceive data" — this is the definition of a Conceptual / Semantic model. The ER model is the prime example.

Note: Relational model (D) is actually an Implementation/Representational model, not conceptual.

Q2

In the Three-Schema Architecture, which schema is responsible for describing physical storage structures and access paths?

Select the best answer.

Correct! The Internal Schema deals with physical reality — B+-trees, indexes, block sizes.
Incorrect. The answer is A — Internal Schema. Conceptual = logical structure; External = user views.

How to solve this:

Remember the three levels with this mental model:

  • • External = what users see (customized views)
  • • Conceptual = what data means (logical structure)
  • • Internal = how data is stored (physical reality)

The question mentions "physical storage structures and access paths" → that's the Internal Schema. Examples include B+-tree indexes, cluster indexes, and block sizes.

Q3

What type of data independence allows you to change the internal schema without affecting the conceptual schema?

Select the best answer.

Correct! Physical Data Independence means changes to how data is stored don't ripple up to the logical level.
Incorrect. Logical independence = changing conceptual without affecting external. The answer is B — Physical Data Independence.

How to solve this:

There are two types of data independence. Use the following memory trick:

Physical

Internal → Conceptual
Change storage, not logic

Logical

Conceptual → External
Change logic, not views

The question says "change internal schema without affecting conceptual" — that's Physical Data Independence (boundary is between internal and conceptual levels).

Q4

A web application has three layers: (1) a browser client with a GUI, (2) an application server running business logic, and (3) a database server. Which DBMS architecture does this represent?

Select the best answer.

Correct! Three layers (client / app server / DB server) = Three-Tier architecture — the standard for modern web apps.
Incorrect. Three distinct layers with a middle application server = C — Three-Tier Client-Server.

How to solve this:

Count the distinct layers in the scenario:

  • • Layer 1: Browser/Client with GUI ← client tier
  • • Layer 2: Application Server with business logic ← middle tier
  • • Layer 3: Database Server ← data tier

3 layers = Three-Tier architecture. The key differentiator from two-tier is the intermediate Application Server that holds business rules. This also adds security through encryption between tiers.

Q5

A database designer adds a new column BirthDate to the EMPLOYEE conceptual schema. The external schemas used by HR and Payroll applications should remain unchanged. What concept enables this?

Select the best answer.

Correct! Logical Data Independence means conceptual schema changes don't force external schema or application changes.
Not quite. A — Logical Data Independence is the ability to change the conceptual schema without affecting external schemas. (D is a mechanism, not the independence property itself.)

How to solve this:

The scenario says: conceptual schema changes (add BirthDate column), but external schemas stay unchanged. This is precisely the definition of Logical Data Independence.

Why not D? External-Conceptual Mapping is the mechanism that enables this (the DBMS updates only the mappings, not the schemas themselves), but the property being described is Logical Data Independence.
