Section 01

Data Models

ISTILAH KUNCI

Data Model

Seperangkat konsep yang mendeskripsikan struktur database dan constraints (batasan) yang harus dipatuhinya. Anggap saja sebagai panduan dasar atau cetak biru untuk database.

ISTILAH KUNCI

Data Model Operation

Operasi untuk menentukan pengambilan (retrieval) dan pembaruan (update) pada database. Bisa berupa fitur bawaan (basic) atau definisi pengguna.

Kategori Data Model

Tingkat Tinggi (High-Level)

Conceptual / Semantic

Dekat dengan bagaimana manusia mempersepsikan data. Menggunakan format Entitas (Entities), Atribut (Attributes), dan Relasi (Relationships).

Contoh: Entity-Relationship (ER) Model

Tingkat Menengah (Mid-Level)

Implementation / Representational

Menyeimbangkan antara sudut pandang pengguna dengan detail penyimpanan komputer.

Contoh: Relational, Network, Hierarchical

Tingkat Rendah (Low-Level)

Physical / Internal

Mendeskripsikan bagaimana data disimpan secara fisik di dalam komputer.

Contoh: Struktur Tree (Pohon), Grafik, B+-trees

⚠ KESALAHAN UMUM

Mahasiswa sering kebingungan dan tertukar antara ER Model (konseptual — untuk merancang) dengan Relational Model (implementasi — untuk database sesungguhnya). ER Diagrams adalah alat desain; tabel relasional adalah implementasinya.

Sejarah Data Model

1964–65 — Network Model Grafik (Graphs)

Diimplementasikan oleh Honeywell (IDS System). Diadopsi oleh CODASYL 1971. Data disimpan sebagai grafik yang saling terkoneksi. Digunakan di IDMS, DMS 1100, IMAGE.

~1965 — Hierarchical Model Pohon (Trees)

IBM + North American Rockwell. Mengembangkan grup IMS. Data disusun dalam hierarki pohon (tree) — dan hanya memiliki sebatas relasi antara orang tua / anak (parent/child relationships).

1970 — Relational Model Tabel (Tables)

Diusulkan oleh E.F. Codd (IBM). Sistem komersil pertamanya sekitar 1981-82. Dan kini sangat mendominasi dunia programming — Oracle, DB2, SQL Server, PostgreSQL.

1980s–90s — Object-Oriented Model Objek (Objects)

Dipelopori program persistent OOP bahasa seperti: C++ (OBJECTSTORE), Smalltalk (GEMSTONE). Dan juga produk O2, ORION (ITASCA), IRIS.

Baru-baru Ini — Object-Relational Model Hybrid

Dimulai dengan Informix Universal Server. Dicontohkan pada keberhasilan tipe di Oracle 10g, DB2, SQL Server. Menyatukan konsep tabel relasi bersama skema OOP.

Relational Model — Konsep Utama

Relasi (Relation) = Tabel Nilai

Sebuah Relation (relasi) adalah konsep matematis didasarkan pada sets / kumpulan nilai
Sebuah relasi dapat dianggap sebagai suatu perhimpunan data per baris / kesatuan barisan (tuples)
Setiap kolomnya dijabarkan sebagai Atribut (Attribute)
Setiap baris (row) dapat memuat tanda identifikasi / unique identifier (key)

Contoh: Tabel STUDENT

Name StudentNumber Class Major
Smith171CS
Brown82CS

← Setiap barisnya adalah sebuah tuple; dan setiap kolom adalah attribute.

💡 ENTITY-RELATIONSHIP (ER) MODEL

Diusulkan oleh P. Chen di tahun (1976). Merupakan alat modeling konseptual untuk menentukan hal-hal wajib pada requirements dari sebuah database. Punya dasar yang sangat berdasar: Entities (Entitas) (obyek kehidupan nyata) serta adanya Relationships (Relasi) (koneksi bersamanya antar Entitas). Sangat baik untuk membantu pemetaan awal pembuatan Relational database.

Catatan: Model ER aslinya TIDAK memakai / menyisipkan unsur *inheritance* dahulu kala.

Section 02

Schema

ISTILAH KUNCI

Database Schema

Deskripsi dari suatu database — yakni melingkupi struktur dan aturannya. Bisa disebut juga sebagai database intension. Berubah sangat jarang (infrequently).

ISTILAH KUNCI

Database State / Snapshot

Data sesungguhnya yang terekam pada satu momen waktu tertentu. Disebut juga extension. Berubah terus menerus setiap data tersebut di-update.

ISTILAH KUNCI

Schema Diagram

Suatu tampilan berupa diagram pada beberapa bagian/aspek database schema — menampilkan nama dari tabel dan berbagai atribut tanpa kehadiran data aslinya.

⚠ KESALAHAN UMUM

Terbalik antara Schema vs State

Schema = cetak biru (stabil, jarang berubah). State = data sesungguhnya (berubah-ubah secara konstan). Jangan samakan struktur dengan isinya!

Three-Schema Architecture

Dibuat untuk tujuan mendukung prinsip program-data independence (kemandirian) dan adanya multiple views (banyak view/sudut pandang) pada perincian data. Memiliki tiga tingkatan layer/abstraksi:

EXTERNAL LEVEL — User Views

External Schema 1

User view A

External Schema 2

User view B

↕ external/conceptual mapping

CONCEPTUAL SCHEMA

Struktur logis & constraints

↕ conceptual/internal mapping

INTERNAL SCHEMA

Struktur penyimpanan fisik

↓

STORED DATABASE

1

Internal Schema (Physical Level)

Menjabarkan bagaimana data secara fisik tersimpan — bentuk/struktur file-nya, susunan indeks data, cara / perlintasan untuk melakukan access paths. Lebih mengedepankan model data secara fisik (kumpulan B+-trees, rupa hash indexes, block sizes dll).

-- Contoh elemen Internal Schema:
Cluster Index on SSN;
B+-tree on Age;
Block size = 100;
2

Conceptual Schema (Logical Level)

Memberikan deskripsi keseluruhan atas struktur rules pada database beserta constraint-nya (batasannya) yang digunakan terhadap semua grup pengguna. Tidak menyebutkan rincian physical data (penyimpanan file fisik). Mengutamakan bagaimana pendefinisian akan arti suatu data.

-- Contoh Conceptual Schema:
EMPLOYEE(SSN, EN, DNO, SALARY, EXP_YEAR, BDATE)
CUSTOMER(Name, Addr, Sex, Age)
3

External Schema (View Level)

Mendeskripsikan rancangan skema turunan (dari Conceptual Schema) sebagai user views berdasarkan kelompoknya. Kelompok user yang dibedakan otomatis akan menyajikan sub-data (subset) dengan penampilannya yang tidak seragam pula (beda user, beda view).

-- Turunan user view dari Conceptual database kita:
CREATE MALE_CUSTOMER(Name, Addr)
  AS VIEW OF CUSTOMER(Name, Addr, Sex, Age)
  WHERE Sex = 'M';

Mencari: "Ayo buatkan daftar list dari para data employee yang pengalamannya lebih dari 5 tahun"

-- External Schema (apa yang diquery pengguna):
SELECT e.ENAME, e.DEPT, e.EXP
FROM   EMP e
WHERE  e.EXP > 5;

-- Dari mana asal "EMP view" ini?, ini diderivasi dari Conceptual Schema:
EMPLOYEE(SSN, EN, DNO, SALARY, EXP_YEAR, BDATE)

-- Dan yang mana ini pun bertumpu pada konfigurasi permesinan (Internal Schema):
Cluster Index on SSN;
B+-tree indexes on DNO, EXP_YEAR, STARTDATE;

Data Independence

Logical Data Independence

Kemampuan untuk memodifikasi Conceptual Schema tanpa berdampak/merusak/mengubah skema views dari User Interface yang dirancang oleh programs yang dijalankan di External Schema.

External schemas tidak berubah
← — — — Batasan LOGIS (LOGICAL boundary) — — —
Conceptual schema bisa dimodifikasi/diubah

Physical Data Independence

Kemampuan yang disematkan untuk bisa menyesuaikan atau mengganti elemen dari Internal schema tanpa memutus/berefek buruk pada struktur Conceptual Schema.

Conceptual schema tidak terpengaruh / tak akan berubah
← — — Batasan FISIK (PHYSICAL boundary) — — —
Internal schema boleh diedit dan dimodifikasi

💡 MODEL MENTAL

Bayangkan layaknya satu bangunan: External schema = apa yang bisa disaksikan oleh sang penghuni kamar/penyewa, Conceptual schema = Arsitek plan pada pembuatan gedung tersebut tersebut, Internal schema = Instalasi pengkabelan / kelistrikannya dan aliran air/logistik. Jika kita hanya ingin mengganti aliran kabelnya, kita kan tak perlu harus membongkar plan blueprint sang arsitek apalagi harus merenovasi hingga penghuni terganggu kan.

Section 03

Komponen DBMS

Bahasa di dalam DBMS

DDL Data Definition Language

Digunakan untuk mendefinisikan schema — perihal struktur tabel dan constraints keseluruhannya. Lebih sering diakses oleh DBA dan database designers.

CREATE ALTER DROP

Tambahan: SDL (Storage Def Language) dipakai untuk internal schema; VDL (View Def Language) untuk external schemas.

DML Data Manipulation Language

Bahasa untuk menarik (retrieve) dan memperbarui (update) data. Bisa jadi tersemat (embedded) di bahasa inti seperti (COBOL, C) atau bahkan dijalankan secara independen (standalone).

SELECT INSERT UPDATE DELETE

HIGH-LEVEL DML (Non-procedural)

Menyebutkan apa ("what") data yang mau diambil, tapi tidak pusing menjabarkan bagaimana ("how")-nya. Berorientasi himpunan set data. Biasa juga disebut bahasa declarative. Contoh: SQL.

LOW-LEVEL DML (Procedural)

Mengeksekusi record/catatan satu-per-satu. Menspesifikasikan secara urut bagaimana (how) untuk mendapatkan / me-retrieve data. Harus di-embed dalam program bahasa secara utuh.

Antarmuka DBMS (Interfaces)

Berbasis Menu (Menu)

Mencari item via dropdown di web

Data Forms (Form)

Sederhana dan user-friendly

Berbasis Grafis (GUI)

Lebih banyak navigasi klik, drag & drop

Natural Language

Diketik dalam bahasa keseharian manusia

Percakapan (Speech/Voice)

Input masuk pakai suara

Antarmuka Milik DBA

Fitur perizinan/manajemen akun

Modul Inti dari DBMS

DDL Compiler

Memproses statements/baris DDL; serta membarui file sistem katalog.

DML Compiler

Meng-*compile*/menerjemahkan bahasa perintah DML yang ada di suatu program asal.

Query Compiler

Bertindak untuk menanggapi pengiriman format instruksi Queries dadakan (ad-hoc) interaktif dari sebagian orang/user kasual.

Run-time Database Processor

Mengeksekusi transactions (transaksi database lengkap) dan penjalanan query level tertinggi yang privilesenya besar.

Stored Data Manager

Mengawasi izin/Akses terhadap info riwayat operasi DBMS di disk memori terpasang.

Concurrency/Backup/Recovery

Berfungsi melindungi aspek properti ACID demi pencegahan data saling tindih atau bila mati listrik.

Algoritma: Bagaimana Sebuah Query Diproses

Langkah demi langkah mengenai alur pemrosesan query di dalam DBMS:

Step 1 / 5

LANGKAH 1 — User Mensubmit Query

Pengguna awam/kasual menulis sebuah string query, ATAU sebuah software program memicu command DML yang dituliskannya.

-- User mengetik interaktif query berikut:
SELECT e.ENAME, e.EXP
FROM   EMP e
WHERE  e.EXP > 5;

LANGKAH 2 — Masuk Modul Compiler DML / Query

Query Compiler memilah dan memilah penulisan string (parsing) lalu memadankannya terhadap Kumpulan Sistem Katalog Data/Dictionary untuk mencocokkan eksistensi table maupun porsi kekuasaan level otorisasi akses (permissions).

-- Compiler mencocokkan pada kamus sistem:
-- Apakah letak table bernama "EMP" ada? ✓
-- Apakah column "EXP" ada isinya? ✓
-- Adakah sang user ini diizinkan nge-SELECT data ini? ✓
-- Lanjut membangun *execution plan*...

LANGKAH 3 — Dieksekusi Run-time Database Processor

Terjemahan kuerasi compiler ini dialihkan kepada Runtime DB Processor buat diproses sedetailnya bekerja-sama dengan modul Stored Data Manager dari sistem.

-- Processor pada Runtime menyusun rancang urut/plan yang optimum:
-- Rencana/Plan: Fokus pakai kolom EXP di index schema tree B+
-- Lacak di direktori index buat record yang angkanya EXP > 5
-- Tarik kecocokan relasi barisannya (tuples)...

LANGKAH 4 — Dioper ke Stored Data Manager

Stored Data Manager menembak akses internal di penyimpanan fisikal DB ini lewat komando aksi program I/O kepada Operating System, memanfaatkan Index pada Internal Schema-nya.

-- Stored Data Manager menembus memori disk fisiknya:
-- Lakukan Read blocks atas letak memori Index-nya
-- Panggil bagian file pages atas record (pages) yang diminta itu
-- Masukkan ke buffer / RAM sesaat sebelum dicap

LANGKAH 5 — Meretur dan Mencetak Output

Relasi tuples/barisan berhasil digenggam dan dihadapkan selaras dengan pandangan permohonan External Schema dari user. Hasil Query tercetak di hadapannya.

-- Hasil diserahkan pada sang penyuruh:
ENAME   | EXP
--------|-----
Alice   | 7
Bob     | 10
Charlie | 6

Section 04

Arsitektur DBMS

Centralized DBMS

Segalanya menumpung dan disatukan di dalam satu wadah memori (single system) — hardware-nya, software DBMS-nya, beserta aplikasinya ditangani menjadi satu.

[PC / Mainframe]
└── DBMS
└── Program App
└── UI Software

2-Tier Client-Server

Pihak Client sekedar mengeksekusi UI dan sedikit program logic. Lalu di sisi Server berjalanlah program DBMS-nya utuh. Dibantu jembatan (API) bernama ODBC (Open DB Connectivity).

[Client]
└── UI + App logic
── ODBC/Network ──
[DBMS Server]

3-Tier Client-Server

Arsitektur paling populer seiring dengan model web modern. Munculnya titik tambahan Application Server di tengah menjadi penghubung/penyimpan otak dari alur bisnis pada webnya.

[Client] Tampilan Web/GUI
── HTTP ──
[App Server] Logic bisnis
── Jaringan SQL ──
[Lumbung Data/ DBMS Server]

💡 FUNGSI LAIN DARI 3-TIER: PERLINDUNGAN (SECURITY)

Model Tiga tingkatan (3-tier) ini menyumbang peningkatan pada sisi sekuritas servernya. Terjadi sebuah kewajiban data encryption sebelum pengiriman dari server dan decrypt di titik/perangkat client penggunanya berlahan-lahan. Application Server seolah-olah ditugaskan membenteng selayaknya firewall yang menghalangi pengguna (user) untuk langsung 'bertemu'/menyentuh isi dalam raw dataset/database terdalam.

Klasifikasi/Pembagian Database

KRITERIA TIPE
Penamaan Data Model-nya Relational (Oracle, DB2), Object-OO (ObjectStore), Network/grafikal (DBTG), Hierarchical/Pohon (IMS)
Pecahan Jumlah Users Single-user (untuk tataran pemakaian level individu/PCs saja), Multi-user (dipakai di enterprise level)
Struktur/Jumlah Sites Server Centralized (Satu tempat), Distributed (Terdistribusi silang R*), Federated (Digabung tapi beda asal vendor: homogeneous/heterogeneous)
Sistem Akses Metode General purpose DBMS (Biasa umum/luas dipakai), Special purpose DBMS (Punya ceruk tujuan penciptaan aplikasi yang sangat sempit/eksotis ditangani DBMSnya)
