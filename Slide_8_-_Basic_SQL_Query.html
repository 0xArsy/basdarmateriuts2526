Pemahaman Mendalam

1. SELECT Statement

Operasi dasar pengambilan data

Mental Model: Bayangkan kamu punya lemari arsip besar (database). SELECT adalah cara kamu membuka laci, memilih folder tertentu, dan mengambil dokumen yang kamu butuhkan berdasarkan kriteria yang kamu tentukan.

SINTAKS DASAR SQL 
SELECT   <attribute list>    -- Kolom yang ingin ditampilkan
FROM     <table list>        -- Tabel sumber data
WHERE    <condition>;        -- Kondisi filter (opsional)

Key Term: Tuple

Tuple adalah satu baris data dalam tabel. SQL membolehkan dua tuple yang identik ada dalam satu tabel — berbeda dengan model relasional formal yang harus unik. Gunakan constraint PRIMARY KEY pada DDL untuk mencegahnya.

Common Pitfall: Duplikasi Data

Tanpa constraint, SQL tidak otomatis menolak duplikasi baris! Jika kamu menyisipkan data yang sama dua kali, keduanya akan tersimpan. Selalu definisikan PRIMARY KEY saat membuat tabel.

2. Cartesian Product (Cross Product)

Operasi penggabungan semua kombinasi tuple

Mental Model: Bayangkan dua set kartu: Kartu A berisi {♠, ♥, ♣} dan Kartu B berisi {1, 2}. Cartesian Product menghasilkan semua kemungkinan pasangan: {♠1, ♠2, ♥1, ♥2, ♣1, ♣2}.

R1 (3 baris)
A    B    C
a1   b1   c1
a2   b2   c2
a3   b3   c3
×
R2 (2 baris)
X    Y
x1   y1
x2   y2
=
3 × 2 = 6 baris
R3 = R1 × R2 (6 baris)
a1 b1 c1 x1 y1
a1 b1 c1 x2 y2
a2 b2 c2 x1 y1
a2 b2 c2 x2 y2
a3 b3 c3 x1 y1
a3 b3 c3 x2 y2
-- Cartesian Product di SQL:
SELECT * FROM R1, R2;
-- Tanpa WHERE = semua kombinasi (berbahaya jika tabel besar!)
Common Pitfall: Lupa WHERE pada Multiple Table

Jika kamu menulis FROM R1, R2 tanpa WHERE, SQL akan menghasilkan Cartesian Product penuh. Jika R1 punya 1000 baris dan R2 punya 1000 baris, hasilnya adalah 1.000.000 baris!

3. Join

Menggabungkan tuple terkait dari dua relasi

Mental Model: JOIN seperti mencocokkan dua daftar hadir: daftar karyawan dan daftar departemen. Kamu hanya pasangkan baris yang memiliki nilai departemen yang sama, bukan semua kemungkinan pasangan.

Key Term: Join Condition

Join condition didefinisikan di dalam klausa WHERE. Ini memfilter hasil Cartesian Product sehingga hanya tuple yang memenuhi kondisi yang digabungkan. Biasanya menghubungkan Primary Key dengan Foreign Key. 

-- JOIN antara EMPLOYEE dan DEPARTMENT
SELECT Fname, Lname, Dname
FROM EMPLOYEE, DEPARTMENT
WHERE Dno = Dnumber;  -- Join condition: FK = PK

-- Contoh: semua karyawan dept Research
SELECT Fname, Lname, Address
FROM EMPLOYEE, DEPARTMENT
WHERE Dname = 'Research' AND Dnumber = Dno;
Key Term: Multi-Table Join

Kamu bisa join lebih dari dua tabel. Tambahkan kondisi untuk setiap pasangan tabel yang dihubungkan:
WHERE A.key = B.ref AND B.key = C.ref

Common Pitfall: Ambiguous Column

Jika dua tabel punya kolom dengan nama sama (misal: Name), kamu HARUS kualifikasi dengan nama tabel: EMPLOYEE.Name, DEPARTMENT.Name.

4. Aliases

Nama alternatif untuk tabel atau kolom

Mental Model: Alias seperti nama panggilan. Kamu bisa refer ke tabel EMPLOYEE sebagai E agar lebih singkat — atau bahkan memakai tabel yang sama dua kali dengan nama berbeda untuk self-join. 

-- Query 8: Karyawan dan nama supervisornya (self-join)
SELECT E.Fname, E.Lname, S.Fname, S.Lname
FROM EMPLOYEE E, EMPLOYEE S
WHERE S.Ssn = E.Super_ssn;

-- Alternatif dengan keyword AS
SELECT E.Fname, E.Lname, S.Fname, S.Lname
FROM EMPLOYEE AS E, EMPLOYEE AS S
WHERE E.Super_ssn = S.Ssn;
Key Term: Self-Join

Self-join terjadi saat kamu perlu menggabungkan sebuah tabel dengan dirinya sendiri. Ini terjadi pada relasi hirarki (supervisor-bawahan). Alias wajib digunakan agar SQL bisa membedakan kedua "salinan" tabel tersebut.

5. WHERE Clause Lanjutan

LIKE, BETWEEN, DISTINCT, ORDER BY

LIKE — Pattern Matching

% = mengganti nol atau lebih karakter apapun

_ = mengganti tepat satu karakter

-- Alamat yang mengandung 'Houston,TX'
WHERE Address LIKE '%Houston,TX%'

-- Nama yang diawali 'Jo'
WHERE Fname LIKE 'Jo%'

-- Nama dengan tepat 4 karakter
WHERE Fname LIKE '____'

BETWEEN & DISTINCT & ORDER BY

-- BETWEEN: gaji antara 30000-40000
WHERE Salary BETWEEN 30000 AND 40000

-- DISTINCT: hapus duplikat hasil
SELECT DISTINCT Salary FROM EMPLOYEE

-- ORDER BY: urutkan hasil
ORDER BY Lname ASC, Salary DESC

-- Asterisk: semua kolom
SELECT * FROM EMPLOYEE WHERE Dno = 5
Key Term: Set Operations

UNION — gabungan, duplikat dihapus
EXCEPT — perbedaan dua set
INTERSECT — irisan dua set
Tambahkan ALL untuk menjaga duplikat.

Common Pitfall: WHERE = TRUE

Jika kamu tidak menuliskan klausa WHERE, kondisinya dianggap TRUE — artinya semua baris dipilih. Pada query multi-tabel, ini menghasilkan Cartesian Product penuh!

Algorithm & Logic Visualization

Visualisasi & Walkthrough

Q0

Retrieve Birth Date & Address of John B. Smith

SELECT   Bdate, Address
FROM     EMPLOYEE
WHERE    Fname = 'John'
  AND    Minit = 'B'
  AND    Lname = 'Smith';

WALKTHROUGH — Klik setiap langkah:

FROM

SQL mengakses tabel EMPLOYEE. Pada tahap ini, semua baris dari tabel dimuat sebagai kandidat — belum ada filter.

WHERE

Tiga kondisi dievaluasi dengan operator AND: Fname = 'John' DAN Minit = 'B' DAN Lname = 'Smith'. Hanya baris yang memenuhi ketiga kondisi sekaligus yang dipertahankan.

SELECT

Dari baris yang lolos filter, hanya kolom Bdate dan Address yang ditampilkan. Kolom lain diabaikan dari output.

Q2

Projects in Stafford — Multi-Table Join

-- Setiap proyek di Stafford: nomor proyek, nomor dept, nama manajer, alamat, dan tanggal lahir
SELECT   Pnumber, Dnum, Lname, Address, Bdate
FROM     PROJECT, DEPARTMENT, EMPLOYEE
WHERE    Dnum = Dnumber          -- Join PROJECT ↔ DEPARTMENT
  AND    Mgr_ssn = Ssn          -- Join DEPARTMENT ↔ EMPLOYEE (manajer)
  AND    Plocation = 'Stafford'; -- Filter tambahan

WALKTHROUGH:

FROM

SQL membuat Cartesian Product dari tiga tabel: PROJECT × DEPARTMENT × EMPLOYEE. Jika masing-masing punya 10, 3, dan 8 baris, hasilnya adalah 10×3×8 = 240 kombinasi sementara sebelum difilter.

JOIN

Kondisi pertama Dnum = Dnumber menghubungkan PROJECT dengan DEPARTMENT. Kondisi kedua Mgr_ssn = Ssn menghubungkan DEPARTMENT dengan karyawan yang menjadi manajernya.

FILTER

Kondisi ketiga Plocation = 'Stafford' lebih lanjut memfilter hanya proyek yang berlokasi di Stafford. Ini adalah selection condition (bukan join condition).

Q8

Self-Join — Karyawan dan Supervisor

-- Tampilkan nama karyawan dan nama supervisor langsungnya
SELECT   E.Fname, E.Lname, S.Fname, S.Lname
FROM     EMPLOYEE AS E, EMPLOYEE AS S
WHERE    E.Super_ssn = S.Ssn;
Cara Kerjanya

Tabel EMPLOYEE dipakai dua kali: E untuk karyawan biasa, S untuk supervisor. Kondisi E.Super_ssn = S.Ssn memastikan kita mencocokkan setiap karyawan E dengan supervisornya di "salinan S". Alias wajib ada agar SQL tidak bingung membedakan keduanya.

Q4A

UNION — Gabungkan Dua Result Set

-- Nomor proyek yang melibatkan karyawan dengan Lname = 'Smith'
-- (sebagai pekerja ATAU sebagai manajer departemen pengendali)

(SELECT DISTINCT Pnumber
 FROM PROJECT, DEPARTMENT, EMPLOYEE
 WHERE Dnum = Dnumber AND Mgr_ssn = Ssn
   AND Lname = 'Smith')

UNION  -- Duplikat otomatis dihapus

(SELECT DISTINCT Pnumber
 FROM PROJECT, WORKS_ON, EMPLOYEE
 WHERE Pnumber = Pno AND Essn = Ssn
   AND Lname = 'Smith');

Soal 1
Konsep Dasar SELECT

Manakah query yang benar untuk menampilkan semua nama depan (Fname) dan nama belakang (Lname) karyawan yang bekerja di departemen nomor 5?

✓ Jawaban: B

Metodologi: Analisis setiap opsi:

  • • A benar dari sisi WHERE, tapi menggunakan * (semua kolom), bukan hanya Fname & Lname.
  • • B ✓ Benar — SELECT menyebut dua kolom yang diminta, FROM EMPLOYEE adalah tabel yang tepat, WHERE Dno = 5 adalah filter yang benar.
  • • C Menggunakan <> berarti "tidak sama dengan 5" — kebalikan dari yang diminta.
  • • D Sintaks tidak valid: tidak ada klausa FROM.

Struktur wajib: SELECT [kolom] FROM [tabel] WHERE [kondisi]

Soal 2
Cartesian Product

Tabel A memiliki 4 baris dan Tabel B memiliki 6 baris. Jika dieksekusi query SELECT * FROM A, B;, berapa banyak baris yang dihasilkan?

✓ Jawaban: C — 24 baris

Metodologi Cartesian Product:

Rumus: |A × B| = |A| × |B|

Setiap baris di A dipasangkan dengan setiap baris di B:
4 baris × 6 baris = 24 kombinasi

Query tanpa WHERE pada multi-tabel = Cartesian Product penuh. Ini mengapa selalu disarankan menambahkan JOIN condition pada WHERE clause!

Soal 3
Aliases & Self-Join

Mengapa aliases E dan S diperlukan dalam query berikut?
SELECT E.Fname, S.Fname FROM EMPLOYEE E, EMPLOYEE S WHERE E.Super_ssn = S.Ssn;

✓ Jawaban: B

Metodologi: Ini adalah kasus Self-Join.

Ketika kamu menulis FROM EMPLOYEE E, EMPLOYEE S, SQL secara konseptual membuat dua "salinan" tabel EMPLOYEE:

  • • E = salinan untuk karyawan biasa
  • • S = salinan untuk supervisor

Tanpa alias, SQL tidak bisa memahami mana kolom Ssn yang dimaksud — milik karyawan atau milik supervisor? Alias membuatnya eksplisit: E.Super_ssn vs S.Ssn.

Soal 4
LIKE & Pattern Matching

Berdasarkan tabel EMPLOYEE berikut, query mana yang menghasilkan karyawan dengan alamat yang mengandung kata "Houston"?

FnameAddress
John731 Fondren, Houston, TX

✓ Jawaban: C

Metodologi Pattern Matching:

  • • A — = mencari string yang persis "Houston" saja. Tidak cocok karena alamat punya karakter lain di depan dan belakang.
  • • B — LIKE 'Houston' tanpa wildcard sama seperti =. Tetap tidak cocok.
  • • C ✓ — '%Houston%': % di depan berarti "ada karakter apapun sebelumnya", % di belakang berarti "ada karakter apapun sesudahnya". Ini menangkap "731 Fondren, Houston, TX" dst.
  • • D — '_Houston_': underscore hanya mengganti 1 karakter. "731 Fondren, Houston, TX" punya banyak karakter di sekitar "Houston".

Latihan Lanjutan — Latihan Slide

Merujuk pada tabel EMPLOYEE dan DEPARTMENT di slide latihan (emp berisi emp_no, emp_fname, emp_lname, dept_no; dept berisi dept_no, dept_name, location), manakah query yang benar untuk menjawab: "Di departemen manakah pegawai bernama depan Matthew bekerja?"

✓ Jawaban: B

Metodologi: Pertanyaan ini membutuhkan data dari DUA tabel berbeda — nama departemen ada di DEPARTMENT, nama karyawan ada di EMPLOYEE.

  • • A — Query multi-tabel tanpa join condition = Cartesian Product! Ini akan menghasilkan semua kombinasi dept_name dengan Matthew, termasuk yang tidak relevan.
  • • B ✓ — Benar. Ada selection condition emp_fname = 'Matthew' DAN join condition EMPLOYEE.dept_no = DEPARTMENT.dept_no. Ini menghubungkan karyawan Matthew ke departemennya yang benar.
  • • C — DEPARTMENT tidak punya kolom emp_fname. Query ini akan error.
  • • D — Hanya menampilkan data karyawan, tidak ada dept_name dari tabel DEPARTMENT.

Dari data slide: Matthew bekerja di dept_no = d3, yaitu departemen "Marketing" yang berlokasi di Dallas.
