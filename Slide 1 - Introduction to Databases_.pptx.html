Conceptual Deep Dive

What is a Database?

A database is a collection of related data — known facts that can be recorded and that have implicit meaning. Think of it as a structured digital filing cabinet where every piece of information is interconnected and purposeful.

Key Terms
Database
An organised collection of interrelated data built for a specific purpose.
Meta-data
Data about data — the schema stored in the catalog (column names, data types, constraints).
Query
A request to retrieve specific data without modifying it.
Transaction
A read/write operation executed as an atomic unit.
Common Pitfall

Not every collection of files is a database! A folder of random spreadsheets lacks the three essential properties: miniworld representation, logical coherence, and specific purpose. Simply storing data ≠ having a database.

Mental Model

Think of a university student record system. The “miniworld” is the university. The data (names, courses, grades) is logically coherent and built specifically to manage academic records — not just dumped in files.

Miniworld (UoD)

Represents some aspect of the real world. Changes in the miniworld are reflected in the database.

Logically Coherent

Data has inherent meaning and relationships — not a random assortment of information.

Specific Purpose

Built for an intended user group and preconceived applications — not created by accident.

Key Term: Universe of Discourse (UoD)

The portion of the real world your database models. A library database’s UoD includes books, members, loans — but not the weather outside. Anything outside the UoD should not appear in your schema.

Common Pitfall

Confusing data with information. Data are raw facts. Information is processed, contextualised data. A database stores data; queries transform it into information.

What is a DBMS?

A Database Management System is a collection of programs that enables users to create and maintain a database. It acts as an intermediary between users/apps and the actual stored data.

Mental Model

A DBMS is like a library management system. Books = data. Catalog = meta-data. Librarian = DBMS — enforcing rules on who borrows what, when, and how many at once.

DBMS Key Functions
→ Define: Specify data types, structures, constraints
→ Construct: Store data on a storage medium
→ Manipulate: Query, update, generate reports
→ Share: Allow multiple users/apps simultaneously

Two distinct groups interact with a database system: those on the scene (who use it) and those behind the scene (who build and maintain it).

Actors on the Scene

Database Administrator (DBA)
Authorises access, monitors use, acquires hardware/software resources.
Database Designers
Identify data to store; choose structures to represent it.
End Users
Naive / Casual / Sophisticated / Standalone — people who query and update data day-to-day.
System Analysts & App Programmers
Determine requirements and implement programs against the DBMS.

Workers Behind the Scene

DBMS Designers & Implementers
Design and build the DBMS software itself (e.g., the team that built PostgreSQL).
Tools Developers
Create optional tools for DB design, performance monitoring, import/export.
Operators & Maintenance
Run and maintain the hardware/software environment for the database system.
Key Distinction

Actors use the database. Workers build and maintain the infrastructure. A DBA sits on the boundary — they use DBMS tools to manage the system on behalf of users.

USE a Database
  • Large business data needing frequent updates
  • Homogeneous records at scale
  • Long-term data relevance
  • Multi-user simultaneous access
DON’T Need One
  • Simple, static, never-changing app
  • Strict real-time constraints
  • Embedded/limited storage systems
  • Single user, no concurrency
CANNOT Use One
  • Data complexity exceeds DBMS modeling limits
  • Users need special ops DBMS can’t handle
Mental Model

Using a DBMS for a 50-row personal shopping list is like using an industrial freight truck to carry a bag of groceries — massive overhead for minimal benefit. Match the tool to the scale of the problem.

Algorithm & Logic Walkthrough

Core SQL operations on the University Database from the slides, with a step-by-step walkthrough.

University Database — Schema Definition (DDL)SQL
-- Create STUDENT table (meta-data / structure definition)
CREATE TABLE STUDENT (
  Name           CHAR(30),
  Student_number CHAR(4),
  Class          INTEGER(1),
  Major          Major_type
);
CREATE TABLE COURSE (
  Course_name    CHAR(10),
  Course_number  CHAR(8),
  Credit_hours   INTEGER,
  Department     CHAR(4)
);
CREATE TABLE GRADE_REPORT (
  Student_number     CHAR(4),
  Section_identifier INTEGER,
  Grade              CHAR(1)
);
Sample Queries & Updates (DML)SQL
-- Q1: Students in 'Database' course Fall 2008 + their grades
SELECT s.Name, gr.Grade
FROM   STUDENT      s
JOIN   GRADE_REPORT gr  ON s.Student_number      = gr.Student_number
JOIN   SECTION      sec ON gr.Section_identifier = sec.Section_identifier
JOIN   COURSE       c   ON sec.Course_number     = c.Course_number
WHERE  c.Course_name = 'Database'
  AND  sec.Semester  = 'Fall'
  AND  sec.Year      = 08;

-- Q2: Prerequisites of the 'Database' course
SELECT p.Prerequisite_number
FROM   PREREQUISITE p
JOIN   COURSE       c ON p.Course_number = c.Course_number
WHERE  c.Course_name = 'Database';

-- U1: Update Smith's class to sophomore (2)
UPDATE STUDENT
SET    Class = 2
WHERE  Name  = 'Smith';
