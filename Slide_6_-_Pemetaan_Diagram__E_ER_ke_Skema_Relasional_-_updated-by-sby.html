SECTION 01

Conceptual Deep Dive

Apa itu Pemetaan (E)ER ke Skema Relasional?

Pemetaan adalah proses transformasi desain konseptual (diagram ER/EER) menjadi skema yang dapat diimplementasikan di sistem basis data relasional. Bayangkan ER diagram sebagai "cetak biru arsitektur" dan skema relasional sebagai "denah teknis konstruksi" — keduanya merepresentasikan hal yang sama, namun di level abstraksi yang berbeda.

Mental Model

Entity type → Tabel (Relation) | Attribute → Kolom | Key attribute → Primary Key | Relationship → Foreign Key atau Tabel baru

KEY TERM

Regular Entity Type

Entity yang memiliki key attribute sendiri dan keberadaannya tidak bergantung pada entity lain. Dipetakan langsung menjadi satu relasi (tabel). Contoh: EMPLOYEE, DEPARTMENT, PROJECT.

KEY TERM

Weak Entity Type

Entity yang tidak memiliki key attribute sendiri dan bergantung pada owner entity. Partial key-nya dikombinasikan dengan primary key pemilik untuk membentuk composite key. Contoh: DEPENDENT bergantung pada EMPLOYEE.

KEY TERM

Foreign Key (FK)

Atribut yang merupakan salinan primary key dari relasi lain, digunakan untuk merepresentasikan hubungan antar entitas. Menjamin referential integrity — setiap nilai FK harus ada di tabel yang dirujuk.

KEY TERM

Surrogate Key

Primary key buatan (artificial) yang dibuat khusus saat entitas-entitas dalam union type memiliki key yang berbeda-beda. Memungkinkan unifikasi referensi dari berbagai superclass ke satu subclass. Contoh: Owner_id pada OWNER.

KEY TERM

Multivalued Attribute

Atribut yang dapat memiliki banyak nilai untuk satu entity. Dipetakan ke relasi baru dengan primary key gabungan dari PK pemilik + nilai atribut tersebut. Contoh: Locations pada DEPARTMENT → DEPT_LOCATIONS.

KEY TERM

Derived Attribute

Atribut yang nilainya diturunkan dari atribut lain (contoh: Number_of_employees dihitung dari jumlah baris). Bisa disimpan (trade-off performa vs. storage) atau tidak disimpan (dihitung on-demand).

⚠ Common Pitfalls
  • M:N tanpa relasi baru — Jika M:N dipetakan seperti 1:N (hanya menambah FK di salah satu sisi), akan terjadi redundansi data masif dan anomali update/delete.
  • Salah penempatan FK pada 1:N — FK harus selalu ada di sisi "N", bukan sisi "1". Menaruh FK di sisi "1" menyebabkan kolom berulang atau multivalued di satu baris.
  • Urutan pemetaan diabaikan — Langkah 1–7 bisa diubah urutan dan hasilnya tetap sama, ASALKAN semua dependensi sudah ada. Namun demi keterbacaan, urutan standar sangat disarankan.
  • Lupa partial key pada Weak Entity — Weak entity wajib menggabungkan PK owner + partial key-nya sebagai composite primary key. Menggunakan partial key saja akan menyebabkan duplikasi data.
  • Opsi EER dipilih salah — Opsi 2 (hanya subclass) hanya valid untuk Total participation. Jika ada partial participation, entitas yang tidak masuk subclass manapun tidak akan punya representasi di tabel.

SECTION 02

Algoritma & Visualisasi Pemetaan ER

Step-by-Step Walkthrough

1

Pemetaan Regular Entity Types

Setiap regular entity type di-mapping ke satu relasi. Semua simple attribute menjadi kolom. Key attribute menjadi primary key.

-- Langkah 1: Regular Entity → Relasi
-- EMPLOYEE (SSN, FName, MInit, LName, BDate, Address, Sex, Salary)
CREATE TABLE EMPLOYEE (
  Ssn       CHAR(9)      PRIMARY KEY,  -- key attribute
  FName     VARCHAR(15),
  MInit     CHAR(1),
  LName     VARCHAR(15),
  BDate     DATE,
  Address   VARCHAR(50),
  Sex       CHAR(1),
  Salary    DECIMAL(10,2)
);

CREATE TABLE DEPARTMENT (
  Dname     VARCHAR(20),
  Dnumber   INT          PRIMARY KEY
);

CREATE TABLE PROJECT (
  Pname     VARCHAR(20),
  Pnumber   INT          PRIMARY KEY,
  Plocation VARCHAR(20)
);

💡 Prinsip: Composite attribute dipecah menjadi kolom terpisah. Derived attribute opsional (tidak disimpan atau disimpan sebagai kolom biasa).

2

Pemetaan Weak Entity Types

Buat relasi baru, salin PK dari owner entity, gabungkan dengan partial key sebagai composite PK. Tambahkan FK constraint ke owner.

-- Langkah 2: Weak Entity → Relasi dengan Composite PK
-- DEPENDENT bergantung pada EMPLOYEE
-- Partial key: Dependent_name
CREATE TABLE DEPENDENT (
  Essn           CHAR(9),       -- disalin dari EMPLOYEE.Ssn
  Dependent_name VARCHAR(15),   -- partial key
  Sex            CHAR(1),
  Bdate          DATE,
  Relationship   VARCHAR(10),

  -- Composite PK: owner_key + partial_key
  PRIMARY KEY (Essn, Dependent_name),
  FOREIGN KEY (Essn) REFERENCES EMPLOYEE(Ssn)
  -- ↑ Referential integrity constraint
);

⚠ Ingat: Partial key saja tidak cukup sebagai primary key karena bisa duplikat lintas owner yang berbeda. Contoh: dua employee bisa punya dependent bernama "Ahmad".

3

Pemetaan Binary 1:1 Relationship Types

Tiga pendekatan tersedia. Pendekatan foreign key adalah yang paling umum: masukkan PK dari salah satu relasi ke relasi lainnya (pilih sisi dengan total participation).

-- Binary 1:1: MANAGES (EMPLOYEE –1:1– DEPARTMENT)
-- Pendekatan 1: Foreign Key (paling umum)
-- DEPARTMENT berpartisipasi total → tambah FK di DEPARTMENT
ALTER TABLE DEPARTMENT
  ADD COLUMN Mgr_ssn        CHAR(9),
  ADD COLUMN Mgr_start_date  DATE,
  ADD FOREIGN KEY (Mgr_ssn)  REFERENCES EMPLOYEE(Ssn);

-- Pendekatan 2: Gabungkan kedua entity + relationship menjadi satu tabel
-- Cocok jika keduanya total participation

-- Pendekatan 3: Cross-reference (buat tabel baru berisi kedua PK)
-- Lebih fleksibel, cocok untuk partial participation

💡 Tips: Relationship attribute (seperti Mgr_start_date) ikut disimpan di relasi yang menerima foreign key.

4

Pemetaan Binary 1:N Relationship Types

Salin PK dari sisi "1" ke relasi sisi "N" sebagai foreign key. Tambahkan relationship attribute jika ada.

-- 1:N Examples dari COMPANY ER Diagram:

-- WORKS_FOR: DEPARTMENT(1) –– N:EMPLOYEE
-- → FK Dno di EMPLOYEE (sisi N) → DEPARTMENT.Dnumber
ALTER TABLE EMPLOYEE
  ADD COLUMN Dno INT,
  ADD FOREIGN KEY (Dno) REFERENCES DEPARTMENT(Dnumber);

-- SUPERVISION: EMPLOYEE(1 Supervisor) –– N:EMPLOYEE(Supervisee)
-- → Recursive relationship, FK ke tabel diri sendiri
ALTER TABLE EMPLOYEE
  ADD COLUMN Super_ssn CHAR(9),
  ADD FOREIGN KEY (Super_ssn) REFERENCES EMPLOYEE(Ssn);

-- CONTROLS: DEPARTMENT(1) –– N:PROJECT
ALTER TABLE PROJECT
  ADD COLUMN Dnum INT,
  ADD FOREIGN KEY (Dnum) REFERENCES DEPARTMENT(Dnumber);
5

Pemetaan Binary M:N Relationship Types

Buat relasi baru (junction table). Salin PK dari kedua entity dan gabungkan sebagai composite PK. Tambahkan relationship attribute jika ada.

-- WORKS_ON: EMPLOYEE(M) –– N:PROJECT
-- → Buat tabel baru WORKS_ON
CREATE TABLE WORKS_ON (
  Essn  CHAR(9),   -- FK → EMPLOYEE.Ssn
  Pno   INT,       -- FK → PROJECT.Pnumber
  Hours DECIMAL(4,1), -- relationship attribute

  PRIMARY KEY (Essn, Pno), -- composite PK
  FOREIGN KEY (Essn) REFERENCES EMPLOYEE(Ssn),
  FOREIGN KEY (Pno)  REFERENCES PROJECT(Pnumber)
);

/* Kenapa tidak bisa seperti 1:N?
   Jika ditaruh FK Pno di EMPLOYEE:
   - Employee A bekerja di Project 1, 2, 3
   - Perlu 3 baris di EMPLOYEE → duplikasi data (anomali!)
   - Atau kolom multi-nilai → melanggar 1NF */

🚨 Critical: M:N selalu menghasilkan tabel baru. Tidak ada cara lain yang tidak menyebabkan redundansi atau pelanggaran normal form.

6

Pemetaan Multivalued Attribute

Buat relasi baru, gabungkan PK owner + nilai atribut sebagai composite PK. Mirip dengan M:N tapi arahnya dari atribut, bukan relationship.

-- DEPARTMENT.Locations adalah multivalued attribute
-- → Buat tabel DEPT_LOCATIONS
CREATE TABLE DEPT_LOCATIONS (
  Dnumber   INT,            -- FK → DEPARTMENT.Dnumber
  Dlocation VARCHAR(20),    -- nilai multivalued attribute

  PRIMARY KEY (Dnumber, Dlocation),
  FOREIGN KEY (Dnumber) REFERENCES DEPARTMENT(Dnumber)
);

-- Contoh data:
-- Dnumber | Dlocation
-- --------+-----------
-- 1       | Houston
-- 1       | Stafford   ← satu dept bisa banyak lokasi
-- 2       | Bellaire
-- 2       | Sugarland
-- 2       | Houston    ← OK! Composite PK mencegah duplikat
7

Pemetaan N-ary (Higher Degree) Relationship Types

Seperti M:N, buat relasi baru yang menyalin PK dari semua entitas yang terlibat. Gabungkan sebagai composite PK (atau pilih sebagian tergantung semantik).

-- SUPPLY: ternary relationship
-- SUPPLIER – SUPPLY – PROJECT, dengan PART juga terlibat
CREATE TABLE SUPPLY (
  Sname     VARCHAR(20),  -- FK → SUPPLIER.Sname
  Proj_name VARCHAR(20),  -- FK → PROJECT.Proj_name
  Part_no   INT,          -- FK → PART.Part_no
  Quantity  INT,          -- relationship attribute

  PRIMARY KEY (Sname, Proj_name, Part_no),
  FOREIGN KEY (Sname)     REFERENCES SUPPLIER(Sname),
  FOREIGN KEY (Proj_name) REFERENCES PROJECT(Proj_name),
  FOREIGN KEY (Part_no)   REFERENCES PART(Part_no)
);
Langkah 1 dari 7
SECTION 03

Pemetaan Model EER ke Skema Relasional

1

Banyak Relasi: Superclass + Subclasses

Buat relasi untuk superclass DAN setiap subclass. Subclass menyimpan PK superclass + atribut spesifiknya.

Semua constraint: ✓
-- Superclass
EMPLOYEE(Ssn, Name, ...)

-- Subclasses
SECRETARY(Ssn→, Typing_speed)
TECHNICIAN(Ssn→, Tgrade)
ENGINEER(Ssn→, Eng_type)
2

Banyak Relasi: Hanya Subclasses

Setiap subclass menyalin atribut superclass. Tidak ada tabel superclass.

Hanya Total Disjoint/Overlap
-- Atribut superclass digabung
CAR(Vehicle_id, Price,
    License_plate_no,
    Max_speed, No_of_passengers)
TRUCK(Vehicle_id, Price,
     License_plate_no,
     No_of_axles, Tonnage)
3

Satu Relasi + Satu Type Attribute

Semua entity (super+sub) digabung dalam satu tabel. Kolom type attribute menandai subtype.

Total/Parsial Disjoint
EMPLOYEE(
  Ssn, Name, ...,
  Job_type, ← type attribute
  Typing_speed,
  Tgrade,
  Eng_type
)
4

Satu Relasi + Banyak Type Attribute

Seperti opsi 3, tapi untuk overlap: tiap subclass punya flag boolean sendiri.

Semua constraint (esp. overlap)
PART(
  Part_no, Description,
  Mflag,  ← boolean flag
  Drawing_no, Batch_no, ...,
  Pflag,  ← boolean flag
  Supplier_name, List_price
)

Perbandingan Opsi Pemetaan Spesialisasi/Generalisasi

Opsi Kapan Digunakan Kelebihan Kekurangan
Opsi 1 Semua constraint Tidak ada NULL values, struktur bersih JOIN diperlukan untuk data lengkap
Opsi 2 Total disjoint/overlap saja Query ke subclass lebih cepat Duplikasi atribut superclass
Opsi 3 Total/parsial disjoint Satu tabel, query simple Banyak NULL untuk atribut subclass lain
Opsi 4 Overlap constraint Mendukung entity di banyak subclass Banyak NULL dan flag boolean

Q1

Pada diagram ER COMPANY, relasi WORKS_ON antara EMPLOYEE dan PROJECT memiliki kardinalitas M:N. Bagaimana cara pemetaan yang tepat ke skema relasional?

A
Tambahkan FK Pno di tabel EMPLOYEE yang menunjuk ke PROJECT
B
Buat tabel baru WORKS_ON(Essn, Pno, Hours) dengan composite PK (Essn, Pno)
C
Tambahkan FK Essn di tabel PROJECT yang menunjuk ke EMPLOYEE
D
Gabungkan EMPLOYEE dan PROJECT menjadi satu tabel besar

📖 Metodologi Penyelesaian

Langkah 5 Algoritma Pemetaan — Binary M:N Relationship Types.

Pilihan A dan C salah karena: jika satu employee bisa bekerja di banyak project, maka FK tunggal di salah satu tabel tidak bisa merepresentasikan kondisi ini tanpa duplikasi baris (melanggar 1NF).

Pilihan D salah karena entitas EMPLOYEE dan PROJECT adalah dua entity type yang berbeda dan harus tetap terpisah.

✅ Pilihan B benar: Tabel junction WORKS_ON dengan composite PK (Essn, Pno) memastikan setiap pasangan (employee, project) hanya ada sekali, dan atribut relationship Hours bisa disimpan.

Q2

Entitas DEPENDENT adalah weak entity yang bergantung pada EMPLOYEE. Partial key DEPENDENT adalah Dependent_name. Apa primary key dari relasi DEPENDENT setelah dipetakan?

A
Hanya Dependent_name
B
Hanya Essn (FK dari EMPLOYEE)
C
Composite key (Essn, Dependent_name)
D
Auto-increment surrogate key baru

📖 Metodologi Penyelesaian

Langkah 2 Algoritma Pemetaan — Weak Entity Types.

Mengapa A salah? Partial key (Dependent_name) bisa sama untuk dependent dari employee berbeda. Misal dua employee berbeda sama-sama punya anak bernama "Budi" → duplikasi PK.

Mengapa B salah? Satu employee bisa punya banyak dependent → Essn saja tidak unik per baris DEPENDENT.

Mengapa D salah? Surrogate key opsional dan tidak menggunakan prinsip pemetaan standar weak entity.

✅ Pilihan C benar: Kombinasi (Essn, Dependent_name) unik — satu employee tidak mungkin punya dua dependent dengan nama yang sama.

Q3

Dalam pemetaan EER untuk spesialisasi dengan constraint Total Overlap, opsi mana yang TIDAK VALID digunakan?

A
Opsi 3: Satu relasi dengan satu type attribute (disjoint assumption)
B
Opsi 1: Banyak relasi dengan superclass dan subclasses
C
Opsi 2: Banyak relasi, hanya subclasses
D
Opsi 4: Satu relasi dengan banyak type attribute (boolean flags)

📖 Metodologi Penyelesaian

Topik: Pemetaan EER — Pilihan berdasarkan constraint

Total Overlap artinya: setiap entity di superclass BISA berada di lebih dari satu subclass sekaligus.

Opsi 3 menggunakan satu type attribute tunggal — ini mengasumsikan entitas hanya bisa masuk ke satu subclass (disjoint). Jika entity bisa masuk ke banyak subclass (overlap), satu kolom type tidak cukup untuk merepresentasikannya.

✅ Pilihan A benar: Opsi 3 tidak valid untuk Total Overlap. Opsi 4 (boolean flags) adalah yang tepat untuk overlap.

Q4

Atribut Locations pada entitas DEPARTMENT adalah multivalued attribute. Apa primary key dari relasi baru DEPT_LOCATIONS?

A
Hanya Dlocation
B
Hanya Dnumber
C
Auto-increment ID baru
D
Composite key (Dnumber, Dlocation)

📖 Metodologi Penyelesaian

Langkah 6 — Multivalued Attribute.

A salah: Lokasi yang sama bisa dimiliki oleh banyak department (Houston bisa ada di dept 1 dan dept 2).

B salah: Satu department bisa punya banyak lokasi (misal dept Research di Houston, Stafford, Houston II).

C tidak optimal: Auto-increment memungkinkan duplikasi (dept 1, Houston bisa muncul dua kali).

✅ D benar: (Dnumber, Dlocation) — kombinasi ini unik karena satu department tidak boleh terdaftar di lokasi yang sama dua kali.

Q5

Pada pemetaan Union Types (kategori), kapan surrogate key diperlukan?

A
Selalu diperlukan dalam setiap union type
B
Ketika superclass-superclass dalam union memiliki primary key yang berbeda-beda
C
Hanya saat jumlah superclass lebih dari 3
D
Surrogate key tidak pernah digunakan dalam pemetaan EER

📖 Metodologi Penyelesaian

Topik: Pemetaan Union Types (Kategori)

Union type menggabungkan entitas dari berbagai superclass yang berbeda-beda. Subclass (kategori) perlu satu primary key yang konsisten.

Contoh dari slide: OWNER bisa berupa PERSON (PK: Ssn), BANK (PK: Bname), atau COMPANY (PK: Cname). Ketiga PK berbeda tipe dan nama → tidak bisa dipakai langsung.

Pengecualian: Jika semua superclass memiliki PK yang sama (seperti CAR dan TRUCK sama-sama pakai Vehicle_id), surrogate key tidak diperlukan.

✅ B benar: Surrogate key dibutuhkan hanya ketika PK dari superclass-superclass berbeda.
