Relational Model Concept

Diperkenalkan oleh Ted Codd dari IBM Research pada tahun 1970. Model ini menggunakan konsep mathematical relation — setiap database direpresentasikan sebagai kumpulan relasi (tabel).

📌 Terminologi Formal vs Non-Formal
Formal (Relational Model)Non-Formal (Sehari-hari)
RelationTable / File
TupleRow / Record
AttributeColumn Header / Field
DomainAll possible column values
Schema of a relationTable definition
State of the relationPopulated table
🔑 Domain D

Domain adalah himpunan nilai-nilai atomic (tidak dapat dibagi lagi) yang valid untuk suatu atribut. Contoh: GPA = bilangan real antara 0.0–4.0. Home_phone = format 7-digit valid dalam suatu area code.

🔑 Relation Schema

Dinyatakan dengan notasi R(A₁, A₂, ..., Aₙ). Contoh: STUDENT(Name, SSN, Home_phone, Address, Office_phone, Age, Gpa) — ini adalah relation dengan degree 7 (7 atribut).

🔑 Relation State r(R)

Adalah subset dari Cartesian product semua domain: r(R) ⊆ dom(A₁) × dom(A₂) × ... × dom(Aₙ). Setiap tuple t adalah list nilai terurut: t = <v₁, v₂, ..., vₙ>.

⚠️ Common Pitfall

Jangan bingung antara Relation Schema (definisi strukturnya) dan Relation State (isi/data aktualnya). Schema bersifat tetap, State berubah seiring waktu ketika data di-insert/delete/update.

2

Karakteristik Relasi

Ordering of Tuples

Relasi didefinisikan sebagai set of tuples — elemen dalam set tidak memiliki urutan. Tabel yang sama dengan baris berbeda urutan tetap merepresentasikan relasi yang sama.

Ordering of Values in Tuple

Secara formal, urutan atribut dalam tuple tidak penting selama korespondensi (atribut → nilai) terjaga. Tuple bisa dipandang sebagai set pasangan <attribute, value>.

Atomic Values (1NF)

Setiap nilai dalam tuple harus atomic. Composite dan multivalued attributes tidak diizinkan (First Normal Form). Contoh: {red, green} sebagai nilai satu atribut ❌.

NULL Values

NULL merepresentasikan nilai yang: (1) tidak diketahui, (2) ada tapi tidak tersedia, atau (3) tidak berlaku untuk tuple tersebut (undefined).

⚠️ Common Pitfall — Multivalued vs NULL

Multivalued attribute seperti Favourite_color = {red, green} harus dipindahkan ke relasi terpisah. NULL bukan berarti 0 atau string kosong — NULL berarti "tidak ada informasi".

3

Relational Model Notation

Nama Relasi

Huruf kapital Q, R, S → nama relasi (schema)

State Relasi

Huruf kecil q, r, s → state (isi) relasi

Tuple

Huruf t, u, v → tuple individual

Dot Notation

R.A → atribut A dari relasi R. t[Ai] → nilai atribut Ai dalam tuple t

-- Contoh akses nilai tuple:
t = <'Barbara Benson', '533-69-1238', '(817)839-8461', ..., 19, 3.25>

-- Akses nilai satu atribut:
t[Name]         = <'Barbara Benson'>

-- Akses subtuple (beberapa atribut sekaligus):
t[Ssn, Gpa, Age] = <'533-69-1238', 3.25, 19>
4

Relational Model Constraints

Constraint adalah batasan pada nilai-nilai aktual dalam database state, yang berasal dari aturan miniworld yang direpresentasikan oleh database.

Implicit / Inherent Constraints

Melekat dalam data model itu sendiri. Contoh: relasi tidak boleh punya duplicate tuples (karena relasi adalah set).

Explicit / Schema-based Constraints

Dapat diekspresikan langsung dalam schema menggunakan DDL. Contoh: PRIMARY KEY, FOREIGN KEY, NOT NULL, UNIQUE.

Application-based / Semantic Constraints

Tidak bisa diekspresikan dalam schema. Diberlakukan lewat program aplikasi. Contoh: "Mahasiswa tidak boleh mengambil lebih dari 24 SKS per semester."

Klasifikasi Integrity Constraints
Key Constraints

Tidak ada dua tuple yang boleh memiliki kombinasi nilai yang sama untuk seluruh atribut key.

Entity Integrity

Nilai primary key tidak boleh NULL. Primary key harus selalu memiliki nilai yang valid.

Referential Integrity

Foreign key harus merujuk ke nilai PK yang ada di relasi yang direferensikan, atau bernilai NULL.

Domain Constraints

Setiap nilai atribut harus berupa nilai atomic dari domain dom(A) yang telah ditentukan.

⚠️ Common Pitfall — Valid vs Invalid State

Valid State = database yang memenuhi semua integrity constraints IC. Invalid State = database yang melanggar minimal satu constraint. Setiap operasi (INSERT/DELETE/UPDATE) harus menjaga database tetap dalam valid state.

Aturan Foreign Key

1. Atribut FK harus memiliki domain yang sama dengan PK yang direferensikan.
2. Nilai FK pada tuple t₁ harus muncul sebagai nilai PK pada suatu tuple di relasi yang direferensikan, ATAU bernilai NULL.

Logic Walkthrough: Cartesian Product & Relation State

Klik setiap langkah untuk memahami bagaimana relasi dibangun secara matematis.

-- Diberikan Relation Schema R(A₁, A₂):
dom(A₁) = { 0, 1 }
dom(A₂) = { a, b, c }

-- Cartesian Product (semua kombinasi yang mungkin):
dom(A₁) × dom(A₂) = {
  <0,a>, <0,b>, <0,c>,
  <1,a>, <1,b>, <1,c>
}  -- 6 kombinasi total (|dom(A₁)| × |dom(A₂)| = 2×3)

-- Satu possible Relation State r(R):
r = { <0,a>, <1,a>, <1,c> }  -- subset dari Cartesian product ✓
Langkah 1 — Definisikan Domain

Tentukan himpunan nilai valid untuk setiap atribut.

Domain adalah himpunan nilai atomic yang valid. dom(A₁) = {0,1} artinya atribut A₁ hanya boleh berisi nilai 0 atau 1. Tidak ada nilai lain yang diizinkan.

Langkah 2 — Hitung Cartesian Product

Kombinasikan semua kemungkinan nilai dari seluruh domain.

Cartesian product menghasilkan semua kombinasi yang mungkin: 2 × 3 = 6 kombinasi. Ini adalah "universe" dari semua tuple yang secara teoritis boleh ada dalam relasi R.

Langkah 3 — Pilih Subset sebagai Relation State

Relation state adalah subset yang mewakili data nyata saat ini.

Tidak semua kombinasi harus ada dalam relasi. r = {<0,a>, <1,a>, <1,c>} adalah state valid karena merupakan subset dari Cartesian product. State berubah seiring operasi INSERT/DELETE.

Langkah 4 — Verifikasi Constraints

Pastikan tidak ada duplicate tuple dan semua nilai dalam domain.

Cek: (a) Semua nilai dalam domain masing-masing atribut ✓, (b) Tidak ada tuple yang duplikat ✓, (c) Jika ada PK, nilainya unik dan tidak NULL ✓. Jika semua terpenuhi → Valid State.

Hierarchy of Keys

Rantai Konsep Key

SK — Superkey

Satu atau lebih atribut yang menjamin keunikan setiap tuple. Bisa mengandung atribut ekstra.

CK — Candidate Key

Minimal superkey — tidak bisa dikurangi atributnya dan tetap unik. Satu relasi bisa punya beberapa candidate key.

PK — Primary Key

Satu candidate key yang dipilih oleh desainer. Ditandai dengan underline dalam notasi schema. Tidak boleh NULL.

AK — Alternate Key

Candidate key yang tidak terpilih sebagai PK. Juga disebut unique key.

FK — Foreign Key

Atribut yang mereferensikan PK di relasi lain. Menjamin referential integrity.

Update Operations & Constraint Violations

INSERT

Memasukkan tuple baru ke dalam relasi R. Dapat melanggar:

  • • Domain Constraints
  • • Key Constraints (duplicate PK)
  • • Entity Integrity (PK = NULL)
  • • Referential Integrity (FK tidak ada di PK)
Default: Tolak insertion
DELETE

Hanya dapat melanggar referential integrity jika tuple yang dihapus direferensikan oleh FK dari tuple lain. Tiga opsi penanganan:

RESTRICT — Tolak penghapusan
CASCADE — Hapus juga tuple yang mereferensikan
SET NULL / SET DEFAULT — Ubah FK menjadi NULL atau default
UPDATE

Memodifikasi nilai atribut tuple. Perlu tentukan kondisi untuk memilih tuple yang dimodifikasi.

Non-key attribute — Biasanya tidak bermasalah
PK / FK — Berpotensi masalah seperti INSERT/DELETE

Transaction Concept

Sebuah transaction adalah sebuah program yang sedang berjalan yang mencakup beberapa operasi database. Transaction harus meninggalkan database dalam keadaan valid/consistent.

OLTP — Online Transaction Processing

Sistem OLTP dapat mengeksekusi ratusan bahkan ribuan transaksi per detik. Contoh nyata: sistem perbankan, e-commerce.

Walkthrough: Constraint Violation pada Operasi

Klik langkah untuk analisis constraint dari soal latihan slide.

-- Data awal: COMPANY database (Fig 3.6)
-- Latihan: Apakah constraints dilanggar?

-- Op 1: INSERT
INSERT INTO PROJECT VALUES (<'ProductA', 4, 'Bellaire', 2>);
-- Pnumber=4 sudah ada? TIDAK → OK. Dnum=2 ada di DEPARTMENT? TIDAK → ❌ Referential Integrity!

-- Op 2: INSERT
INSERT INTO WORKS_ON VALUES (<'677678989', null, '40.0'>);
-- Essn='677678989' ada di EMPLOYEE? TIDAK → ❌ Referential Integrity!
-- Pno=NULL adalah bagian dari PK(Essn,Pno) → ❌ Entity Integrity!

-- Op 3: DELETE
DELETE FROM WORKS_ON WHERE Essn = '333445555';
-- WORKS_ON tidak direferensikan FK dari relasi lain → ✓ Aman

-- Op 4: DELETE
DELETE FROM EMPLOYEE WHERE Ssn = '987654321';
-- Ssn ini direferensikan oleh: DEPARTMENT.Mgr_ssn, EMPLOYEE.Super_ssn
-- → ❌ Referential Integrity! (Cascade/Restrict/SetNull)

-- Op 5: UPDATE
UPDATE EMPLOYEE SET Super_ssn = '943775543'
WHERE Ssn = '999887777';
-- Super_ssn='943775543' ada di EMPLOYEE.Ssn? TIDAK → ❌ Referential Integrity!
Op 1 — INSERT PROJECT

INSERT <'ProductA', 4, 'Bellaire', 2> into PROJECT

Analisis: Pnumber=4 belum ada → Key constraint OK. Dnum=2 → harus ada di DEPARTMENT.Dnumber. Cek: DEPARTMENT hanya punya Dnumber={5,4,1}. Dnumber=2 tidak ada → ❌ Melanggar Referential Integrity! Insert ditolak.

Op 2 — INSERT WORKS_ON

INSERT <'677678989', null, '40.0'> into WORKS_ON

Analisis dua pelanggaran: (1) Essn='677678989' tidak ada di EMPLOYEE.Ssn → ❌ Referential Integrity. (2) Pno=null padahal Pno adalah bagian dari composite PK(Essn,Pno) → ❌ Entity Integrity. Insert ditolak.

Op 3 — DELETE WORKS_ON

DELETE WORKS_ON WHERE Essn = '333445555'

Analisis: DELETE hanya bisa melanggar Referential Integrity jika tuple yang dihapus direferensikan sebagai PK oleh FK lain. Cek: tidak ada relasi yang mempunyai FK menunjuk ke WORKS_ON → ✓ Aman! Operasi berhasil.

Op 4 — DELETE EMPLOYEE

DELETE EMPLOYEE WHERE Ssn = '987654321'

Analisis: Ssn=987654321 direferensikan oleh: DEPARTMENT.Mgr_ssn dan EMPLOYEE.Super_ssn (beberapa karyawan punya Super_ssn ini). → ❌ Referential Integrity dilanggar! Harus pilih: RESTRICT (tolak), CASCADE (hapus semua yang mereferensikan), atau SET NULL.

Op 5 — UPDATE SUPERSSN

UPDATE EMPLOYEE SET Super_ssn='943775543' WHERE Ssn='999887777'

Analisis: Super_ssn adalah FK yang mereferensikan EMPLOYEE.Ssn. Nilai '943775543' harus ada di kolom Ssn EMPLOYEE. Cek data: tidak ada SSN=943775543 → ❌ Referential Integrity! Update ditolak.

Interactive Practice Suite

Soal 1 / 5

Diketahui relasi R(A₁, A₂) dengan dom(A₁) = {0,1} dan dom(A₂) = {a,b,c}. Berapa jumlah tuple yang ada dalam Cartesian product seluruh domain?

Metodologi: Cartesian product dari dua domain dihitung dengan: |dom(A₁)| × |dom(A₂)| = 2 × 3 = 6. Hasilnya: {<0,a>, <0,b>, <0,c>, <1,a>, <1,b>, <1,c>}. Relation state adalah subset dari 6 kombinasi ini — bisa berisi 0 hingga 6 tuple.
Soal 2 / 5

Tabel R1 memiliki nilai multivalued {b1, b2} di kolom B. Tabel R2 memiliki dua tuple identik. Tabel R3 memiliki nilai NULL di kolom AGE. Manakah yang merupakan relasi valid dalam relational database?

Metodologi: R1 melanggar 1NF (First Normal Form) — nilai {b1,b2} adalah multivalued, tidak atomic. R2 melanggar key constraint — dua tuple identik tidak boleh ada dalam relasi (relasi adalah set). R3 valid: semua nilai atomic, tidak ada duplicate, dan NULL adalah nilai yang sah untuk atribut yang tidak berlaku (misal: umur yang tidak diketahui).
Soal 3 / 5

Pada relasi EMPLOYEE, atribut Super_ssn adalah foreign key yang mereferensikan EMPLOYEE.Ssn (self-referencing). Jika kita menghapus tuple EMPLOYEE dengan Ssn='333445555', dan ada 4 tuple WORKS_ON dengan Essn='333445555', apa yang terjadi dengan constraint?

Metodologi: DELETE hanya bisa melanggar Referential Integrity. Ssn='333445555' direferensikan oleh: (1) WORKS_ON.Essn — 4 tuple, (2) mungkin EMPLOYEE.Super_ssn dari karyawan lain. Sehingga penghapusan melanggar referential integrity. Solusinya: RESTRICT (tolak), CASCADE (hapus WORKS_ON terkait), atau SET NULL (ubah Essn=NULL — tapi ini juga melanggar entity integrity WORKS_ON karena Essn adalah PK!).
Soal 4 / 5

Pada skema basis data ORDER (Order#, Odate, Cust#, Ord_Amt), ORDER_ITEM (Order#, Item#, Qty), ITEM (Item#, Unit_price), SHIPMENT (Order#, Warehouse#, Ship_date), WAREHOUSE (Warehouse#, City). Berapa banyak foreign key yang ada?

Metodologi — identifikasi FK satu per satu:
1. ORDER.Cust# → CUSTOMER.Cust#
2. ORDER_ITEM.Order# → ORDER.Order#
3. ORDER_ITEM.Item# → ITEM.Item#
4. SHIPMENT.Order# → ORDER.Order#
5. SHIPMENT.Warehouse# → WAREHOUSE.Warehouse#
Total = 5 foreign key. Strategi: setiap atribut yang bukan PK sendiri tapi merujuk ke PK relasi lain adalah FK.
Soal 5 / 5

Manakah pernyataan yang BENAR mengenai perbedaan antara Superkey, Candidate Key, dan Primary Key?

Metodologi: Hierarki key: Superkey ⊃ Candidate Key ⊃ Primary Key.
(A) ❌ Justru sebaliknya — candidate key adalah minimal superkey, artinya lebih kecil/sama dengan superkey.
(B) ❌ Primary key TIDAK BOLEH NULL (Entity Integrity Constraint).
(C) ❌ Satu relasi bisa memiliki beberapa candidate key (contoh: CAR punya License_number DAN Engine_serial_number).
(D) ✅ BENAR — Candidate key adalah superkey yang minimal (tidak bisa dikurangi atributnya). Setiap CK memenuhi syarat superkey, tapi superkey yang bukan minimal bukan CK.
